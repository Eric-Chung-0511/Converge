## 第一部分：核心觀念 — 為什麼需要蒙地卡羅

### 傳統 CPM 的問題

傳統排程（CPM，Critical Path Method）給每個活動**一個**工期數字，前後推算出**一個**完工日。問題是：工期不是一個數字，是一個**範圍**。基礎開挖「最快 8 天、通常 10 天、下雨可能 20 天」——CPM 只能填一個 10。

CPM 的完工日因此隱含兩個系統性的樂觀偏差：

1. **偏態偏差（skew bias）**：工程活動的工期分布幾乎都是右偏的——提早有極限（最快也要 8 天），延誤沒有極限（可以拖很久）。右偏分布的**平均值 > 最可能值**，而 CPM 用的是最可能值（mode），所以光是這一點，真實的期望完工就比 CPM 晚。
2. **匯流偏差（merge bias）**：多條平行路徑匯合到同一個節點時，要**全部**完成才能往下走。每條路徑各有 50% 機率準時，四條獨立路徑全部準時的機率只有 0.5⁴ ≈ 6%。CPM 完全看不到這件事。

### 蒙地卡羅在做什麼

很簡單的一句話：**把這個專案「虛擬地執行」一萬次**。每一次（iteration）：

1. 每個活動從它的機率分布中抽一個工期（例如三角分布 (8, 10, 20) 抽出 13.2 天）。
2. 用 CPM 邏輯（前後關係）算出這一次的完工日。
3. 記錄下來。

跑完一萬次，就得到一萬個完工日，構成**完工日的機率分布**。之後所有的報表（S-curve、P50、P80、criticality）都只是對這一萬個數字做統計。

類比：CPM 是「假設每場比賽都打出平均表現，預測球隊戰績」；蒙地卡羅是「把整個賽季模擬一萬遍，看戰績的分布」。

## 第二部分：操作流程（一步一步）

### 啟動

```bash
cd converge
streamlit run app/streamlit_app.py
```

### Step 1 — 載入或建立資料（Data Input 分頁）

最快的方式：側欄 **Load Example** 選 `DC Commissioning (MEP → IST)` → 按 Load Example。

自己建資料時，左表是**活動表（Activity Register）**：

| 欄位 | 意義 | 注意 |
|---|---|---|
| id | 活動代號 | 不可重複 |
| predecessors | 前置活動，分號分隔 | `A1;A2`。支援 lag 與 SS：`A1:FS:3`、`A1:SS:5` |
| dist_type | 分布類型 | triangular / pert / uniform / normal |
| param_a, m, b | 三點估計 | a=最樂觀, m=最可能, b=最悲觀（normal 時 a=平均, m=標準差） |

右表是**風險驅動因子表（Risk Driver Register）**：

| 欄位 | 意義 |
|---|---|
| probability | 這個風險在單次模擬中發生的機率（0.35 = 35% 的 iteration 會發生） |
| impact_a/m/b | 發生時的工期**乘數**範圍（1.0–1.15–1.40 = 不變到 +40%） |
| assigned_activities | 受影響的活動，分號分隔 |

也可以用 CSV / Excel 匯入（側欄有範本可下載）。

### Step 2 — 設定模擬參數（側欄）

- **Iterations (N)**：預設 10,000。N 越大曲線越平滑，誤差以 σ/√N 速度縮小（N 變 4 倍，誤差減半）。
- **Sampling method**：Random 或 LHS（見第四部分）。
- **Random seed**：固定 seed = 同樣的輸入永遠得到同樣的結果。Demo 時務必固定，才能重現。
- **Calendar dates（選用）**：勾選後輸入專案起始日，結果會多一張日曆日對照表。注意這只是呈現層，引擎一律以工作天計算。

## 第三部分：每張報表怎麼看

### 3.1 S-Curve（CDF）— 最重要的一張圖

橫軸是完工日，縱軸是「**在這個日期（含）之前完工的機率**」。曲線上任一點 (x, y) 讀作：「有 y 的機率在第 x 個工作天前完工」。

**P 值（percentile）的意義：**

- **P50**（中位數）：50% 機率趕得上。一半的模擬比它早，一半比它晚。
- **P80**：80% 機率趕得上。**業界向業主承諾日期的常用標準**——意思是「我們有八成把握」。
- **P90**：90% 機率。更保守，常用於合約罰則重大或對外公告的日期。
- **P10**：只有 10% 機率達成——通常用來標示「最樂觀也不過如此」。

**為什麼業界用 P80 不用 P50？** P50 是擲硬幣——承諾 P50 日期等於有一半機會跳票。P80 在「合理的緩衝」與「不至於保守到喪失競爭力」之間取得平衡。P50 到 P80 之間的距離就是**排程風險準備（schedule contingency）**的合理大小，而且它是算出來的，不是拍腦袋的 10%。

**怎麼跟業主講這張圖：**「灰色虛線是合約排程的日期，它落在曲線的 12% 位置——意思是照目前的計畫，12 次裡只有 1-2 次趕得上。如果要八成把握，要看 P80，比合約日期晚 23 個工作天。」

**P(finish by target)**：在 target 欄輸入目標日（工作天），會直接顯示如期完工機率——業主真正想問的就是這個數字。

### 3.2 Histogram（PDF）

一萬次完工日的次數分布。重點看：

- **右偏的長尾**：右尾比左尾長很多 = 延誤的空間遠大於提早的空間。
- **峰的位置 vs CPM 線**：CPM 通常落在分布的左側（樂觀側），不在中間。

### 3.3 Merge Bias 分頁 — 本 App 的招牌

這頁把「模擬平均比 CPM 晚多少」拆成兩個成分（很多商業工具混為一談）：

```
CPM (mode 基準)  ── +偏態偏移 ──>  T(E[d])  ── +真正的 merge bias ──>  模擬平均 E[T]
```

- **偏態偏移（mean shift）**：把每個活動換成「平均工期」重新做一次確定性 CPM，得到 T(E[d])。它和原 CPM 的差距來自 mean > mode（右偏）加上風險因子的期望抬升。**就算只有一條路徑也會有這一段。**
- **真正的 merge bias（Jensen gap）**：模擬平均 E[T] 和 T(E[d]) 的差距。數學上由 Jensen 不等式保證 ≥ 0（完工時間是工期的凸函數），而且**只有平行路徑匯流時才大於零**——串聯鏈上它精確為零（凸函數退化為線性）。

**檢驗自己有沒有懂的測試**：Hulett Case 1（單路徑）跑出來 merge bias = 0.000、偏態偏移 = 16.7 天；Case 2（加一條一模一樣的平行路徑）CPM 完全不變，但平均晚了 9.4 天，**全部**來自 merge bias。如果能把這兩個案例講清楚，就真的懂了。

**實務含義**：資料中心調試（commissioning）是教科書級的 merge 場景——配電、冷卻、發電機、BMS 四條 L3→L4 路徑全部完成才能開始 L5 IST。每條路徑各自「大致準時」，IST 開始日卻系統性地晚——這就是 energization / RFS 日期老是樂觀的數學原因。

### 3.4 Criticality & Tornado 分頁

**Criticality Index（CI）**：每個活動「在多少比例的模擬中位於關鍵路徑上」。CPM 的關鍵路徑是 0/1（在或不在）；CI 是機率（這個活動有 64% 的情況在拖累整個專案）。

**核心洞察——CI 經常與 CPM 關鍵路徑不一致**：一個有 5 天浮時（float）的活動，如果它的不確定性範圍是 ±15 天，它「吃掉浮時變成關鍵」的機率非常高。Hulett Case 3：CPM 說 Path A 關鍵（零浮時），但模擬顯示有 5 天浮時的 Path B 才是真正的風險主角（69% vs 31%）。**只盯著 CPM 浮時管理專案，會盯錯對象。**

表中的 ★ 標記 CPM 關鍵路徑——對照 CI 欄就能看出兩者的分歧。

**Tornado（敏感度）**：用 Spearman 等級相關衡量「每個活動的工期」與「總完工日」的關聯強度，由大到小排。最上面的活動就是「最值得花管理注意力的活動」——縮小它的不確定性（提早採購、預先測試、加派人力）對完工日分布的改善最大。

CI 和 Tornado 的差別：CI 問「它多常在關鍵路徑上」，敏感度問「它的變動多大程度帶動完工日變動」。一個工期很穩定的活動可以永遠在關鍵路徑上（CI 高）但敏感度低——因為它不動。

### 3.5 Risk Drivers & Correlation 分頁

**為什麼不直接輸入相關係數？** 傳統做法是拍一個「活動間相關係數 0.3」塞進模型。問題：(1) 沒人知道 0.3 從哪來、答辯不了；(2) 一個全域係數對所有活動對一視同仁，沒有鑑別力。

**Risk Driver Method（Hulett）**：改為建模「風險事件」本身——「供應商延誤，35% 機率發生，發生時設備類活動工期 ×1.15~1.40」。模擬時每次抽「發生/不發生」，發生就把**同一個**乘數套到所有被指派的活動上。共享風險的活動自然同漲同跌——**相關性是結構的產物，不是輸入的假設**。

**Implied Correlation Matrix 怎麼看**：兩個活動共享的 driver 越多、各自獨立的 driver 越少，相關係數越高。只共享一個 driver 且無其他風險 → 接近 1.0；完全不共享 → 接近 0。這張矩陣是從模擬樣本**算出來**的，證明相關性確實湧現了。

### 3.6 Validation Bench 分頁 — 信任的來源

被質疑「你怎麼知道你的引擎是對的」時，就開這頁。三層證明：

1. **解析一致性**：三角分布的平均/變異數有閉式解，模擬值必須在理論容差內吻合 → 證明抽樣正確。
2. **Merge bias vs 數值積分**：E[max(X₁,X₂)] 用 scipy 數值積分算出參考值，模擬必須在 0.5 天內吻合 → 證明網路計算正確。
3. **收斂速率**：蒙地卡羅標準誤理論上以 σ/√N 衰減，log-log 圖上斜率應為 −0.5。實測斜率落在容差內 → 這是最強的整體性證明，抽樣有偏、RNG 壞掉、向量化寫錯任何一項都會讓它失敗。

底下是 **Hulett (1996) 發表基準**的「computed vs published」對照，附原文連結——審閱者可以自己打開原文核對數字。

### 3.7 Distribution Comparison 分頁 — PERT 的陷阱

PERT 和三角分布吃同樣的三點估計 (a, m, b)，但 PERT 把標準差實質固定在約 (b−a)/6，**不管偏態多嚴重**。對高度右偏的活動（如 a=10, m=12, b=60），PERT 的尾部機率比三角分布低得多——**系統性低估尾部風險**。

實務守則：用 PERT 時要自覺「我選了一個比較樂觀的尾部假設」。對真正危險的活動（許可審批、首次執行的工序），值得用三角分布或加大 b 值再看一次。這頁可以並排比較同一組三點估計下兩種分布的差異。

## 第四部分：背後的數學（白話版）

**Jensen 不等式**：對凸函數 f，E[f(X)] ≥ f(E[X])。max 是凸函數，所以「先取期望再取 max」（CPM 的做法）≤「先 max 再取期望」（真實情況）。直覺版：四條路徑只要**任何一條**爆掉，IST 就延後——「最慢的那個」的期望值永遠比「每條都取平均後的最慢」更悲觀。

**中央極限定理（CLT）**：長串聯路徑的總工期 = 許多獨立隨機工期之和 → 趨近常態分布。這就是為什麼單一路徑的完工分布看起來像鐘形，而多路徑匯流後的分布會右偏。

**σ/√N**：模擬估計的標準誤 = 母體標準差 ÷ √迭代數。要把誤差減半，迭代數要 ×4。這也說明為什麼 N=10,000 通常夠：對 σ≈15 天的專案，平均值的誤差已經只剩 0.15 天。

**LHS（拉丁超立方抽樣）**：純隨機抽樣會「運氣不好時某段區間都沒抽到」；LHS 把 [0,1] 切成 N 層、每層強制抽一點，輸入空間覆蓋更均勻 → 同樣的 N 收斂更快、尾部估計更穩。預設可先用 Random（最標準、最好解釋），需要比較時切 LHS。

**為什麼模擬時要拿掉強制日期（constraint dates）**：風險分析的目的就是檢驗合約日期到底可不可行。如果每次迭代都強制「不得晚於 X」，模擬被迫成功，分析就失去意義。所以引擎只看邏輯與工期。這是 QRA 的基本素養題。

## 第五部分：面試應答題庫

**Q：你沒有實際做過 QRA，為什麼我要相信你懂？**
A：所以我把工具做到「可被證偽」的程度——引擎對解析解、數值積分、理論收斂速率三層驗證，並重現 Hulett 1996 的發表結果（附原文連結可核對）。我能展示的不是年資，是我知道這門分析的每個數字為什麼是對的。

**Q：CPM 日期和模擬平均的差距就是 merge bias 嗎？**
A：不是，這是常見的混淆。差距有兩個來源：mean > mode 的偏態偏移（單路徑也有），和平行路徑匯流的 Jensen gap（只在 merge 時出現）。我的工具把兩者分開量化——串聯鏈上量到的 merge bias 精確為零，這本身就是引擎正確性的證據。

**Q：為什麼用 P80 承諾而不是 P50？**
A：P50 等於擲硬幣。P80 是業界平衡點：對業主有八成把握，又不會保守到失去競爭力。而且 P50→P80 的距離給出 contingency 的「計算依據」，取代拍腦袋的百分比。

**Q：活動之間的相關性你怎麼處理？**
A：不輸入相關係數——一個全域係數沒有鑑別力也無法答辯。我用 Hulett 的 Risk Driver Method：建模風險事件（機率 × 乘數 × 指派活動），相關性從共享風險的結構中湧現，而且可以輸出 implied correlation matrix 驗證。

**Q：PERT 跟三角分布選哪個？**
A：PERT 平滑、業界慣用，但它把標準差固定在約 (b−a)/6，對高偏態活動會低估尾部風險。關鍵活動我會雙分布跑一次看尾部差異——知道假設在哪裡咬人，比選哪個更重要。

**Q：迭代幾次才夠？**
A：誤差以 σ/√N 收斂。我的 app 有收斂視圖實測這個速率（log-log 斜率 −0.5）。一般 10,000 次對平均值已綽綽有餘，尾部百分位數要更穩可上 50,000 或開 LHS。

**Q：為什麼模擬中不保留 P6 裡的 constraint dates？**
A：風險分析的目的就是檢驗那些日期可不可行；強制它們成立會讓模擬失去意義。引擎只用邏輯與工期。

### 5 分鐘 demo 腳本

1. **(30秒)** 載入 DC Commissioning 範例：「四條調試子系統路徑匯流到 IST——資料中心交付最經典的風險結構。」
2. **(60秒)** Run → S-curve：「CPM 日期落在 8%——12 次裡 1 次趕得上。P80 在這裡，差距就是該提的 contingency。」
3. **(90秒)** Merge Bias 分頁：「差距不是一筆帳——16 天是偏態、5 天是純匯流效應，分開來才知道對策：偏態靠改估計，merge bias 靠解耦路徑或提早串測。」
4. **(60秒)** Criticality：「CPM 說關鍵的是這條，模擬說 64% 的情況拖累專案的是那條 5 天浮時的——管理注意力該放哪，一目了然。」
5. **(60秒)** Validation：「最後，這些數字為什麼可信——解析解、數值積分、σ/√N 收斂、加上 Hulett 1996 發表結果的重現，原文連結在這。」
