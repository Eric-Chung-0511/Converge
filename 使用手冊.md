## 1. 產品概述

**Converge** 是一套本地端執行的排程風險分析（Schedule Risk Analysis，SRA）工具，核心為向量化蒙地卡羅引擎。它專為有實務 QRA（Quantitative Risk Analysis）需求的專案控制師、規劃師及資料工程師設計。

### 三大差異化能力

| 能力 | 說明 |
|------|------|
| **合併偏差量化** | 當多條平行路徑在同一節點匯合時，蒙地卡羅結果系統性地比 CPM 預測晚。Converge 量化這個差距並以 Jensen 不等式給出數學保證。 |
| **關鍵性指數 ≠ CPM 要徑** | CPM 標記的「零浮動」活動，不一定是最常驅動完工風險的活動。Converge 顯示每個活動在幾趴的迭代中真正位於最長路徑上。 |
| **Hulett 風險驅動因子法** | 相關性由共享風險驅動因子自然湧現，不需手動輸入相關係數。同一驅動因子影響的所有活動，在模擬中同步延長。 |

### 適用場景

- 資料中心、任務關鍵型設施（MEP → 調試）的排程風險評估
- EPC 項目里程碑可行性分析
- 向業主溝通「CPM 日期為何只有 X% 把握」
- 學習與教學：合併偏差、蒙地卡羅收斂性、PERT 假設的限制

## 5. 資料輸入：活動表格

在「**Data Input**」分頁的左半部，有一個互動式活動表格。

### 欄位說明

| 欄位 | 說明 | 範例 |
|------|------|------|
| `id` | 活動的唯一識別碼 | `MEP`、`L3_PWR` |
| `name` | 活動名稱（中英文均可） | `MEP & E&I Installation` |
| `predecessors` | 前置活動 ID，多個以分號分隔 | `L3_PWR;L3_COOL` |
| `dist_type` | 持續時間的機率分布類型 | `triangular`、`pert`、`uniform`、`normal` |
| `param_a` | 分布下限（三角/PERT = 最樂觀值；Normal = 平均值） | `30` |
| `param_m` | 分布眾數（三角/PERT = 最可能值；Normal = 標準差） | `40` |
| `param_b` | 分布上限（三角/PERT = 最悲觀值） | `60` |

> **注意**：所有時間單位為**工作天**（working days），從項目開始日（第 0 天）起算。

### 分布類型說明

#### `triangular`（三角分布）

三點估計法的基礎分布。

- 參數：`a`（最小）、`m`（最可能）、`b`（最大），必須滿足 `a ≤ m ≤ b`
- CPM 時程使用**眾數 m**
- 適用於：一般活動時程估算，需要精確反映尾部風險時
- 數學：平均值 = (a + m + b) / 3

```
範例：某安裝工作 Triangular(20, 30, 50)
  最樂觀 = 20 工作天
  最可能 = 30 工作天（CPM 用這個值）
  最悲觀 = 50 工作天
  模擬平均 = (20+30+50)/3 = 33.3 工作天
```

#### `pert`（Beta-PERT 分布）

專案管理界最常用的三點估計分布，對眾數給予更高的權重（預設 λ=4）。

- 參數：同三角分布
- 比三角分布更平滑、呈鐘形
- **重要限制**：PERT 的有效標準差 ≈ (b-a)/6，對高度偏態活動會**低估尾部風險**
- 適用於：大多數標準 QRA 報告（與業界慣例一致）

```
公式：平均值 = (a + 4m + b) / 6
範例：PERT(20, 30, 50) → 平均值 = (20+120+50)/6 = 31.7 天
對比三角(20, 30, 50) → 平均值 = 33.3 天（三角分布較保守）
```

#### `uniform`（均勻分布）

- 參數：`a`（下限）、`b`（上限）
- 等機率分布，適用於完全不確定的情況
- 僅需填寫 `param_a` 和 `param_b`，`param_m` 不使用

#### `normal`（截斷正態分布）

- 參數：`param_a` = 平均值 μ，`param_m` = 標準差 σ
- 在 0 處截斷（時程不能為負）
- 適用於：演示中央極限定理；或當歷史資料支持正態假設時

### 如何新增活動

點擊表格底部的「**+ 新增一列**」，直接輸入欄位值。Streamlit 的 `data_editor` 支援直接在儲存格內點擊編輯。

### 如何刪除活動

在表格最左側的核取方塊打勾後，按鍵盤 `Delete` 或點擊表格右上角的刪除按鈕。

### 注意事項

1. `id` 必須唯一，不可重複
2. `predecessors` 中列出的 ID 必須已存在於活動表格中
3. 不可有循環相依（例如 A → B → A）
4. 至少需要一個沒有前置活動的「起始」活動
5. **不要設定強制日期限制**：Converge 故意忽略日期限制。風險分析的目的就是測試計畫日期是否可行；若強制日期限制，模擬結果將失效。

## 6. 資料輸入：風險驅動因子表格

在「**Data Input**」分頁的右半部，有風險驅動因子（Risk Driver）表格。

### 欄位說明

| 欄位 | 說明 | 範例 |
|------|------|------|
| `id` | 驅動因子唯一識別碼 | `RD-VENDOR` |
| `name` | 風險名稱 | `Vendor Equipment Delay` |
| `probability` | 發生機率（0.0 到 1.0） | `0.35` 表示 35% 機率 |
| `impact_type` | 衝擊倍數的分布類型 | `triangular` 或 `uniform` |
| `impact_a` | 衝擊倍數下限（1.0 = 無影響） | `1.0` |
| `impact_m` | 衝擊倍數眾數 | `1.15` 表示延長 15% |
| `impact_b` | 衝擊倍數上限 | `1.40` 表示最多延長 40% |
| `assigned_activities` | 受影響的活動 ID，分號分隔 | `L3_PWR;L3_COOL;L3_GEN` |

### 運作邏輯

每次迭代，每個風險驅動因子獨立進行：

1. **發生判定**：以 `probability` 為參數做伯努利抽樣（如 35%，代表約每 100 次有 35 次發生）
2. **衝擊抽樣**：若發生，從 impact 分布中抽取一個倍數（例如抽到 1.23，代表延長 23%）
3. **同步套用**：**同一個倍數**套用到所有被指定的活動

**關鍵結果**：被同一驅動因子指定的活動，它們的時程在這次迭代中**同步變動**，因此形成正相關。這就是「相關性作為結構輸出」的核心機制。

### 設計原則

- `impact_a` 應設為 1.0（不發生影響的基準）
- `impact_b` 代表最壞情況下的延長倍數
- 若某個驅動因子同時影響多個子系統（如「設備供應商延遲」影響所有 L3 測試），請在 `assigned_activities` 中列出所有相關活動
- 驅動因子的影響為**乘法**：若活動原本計畫 30 天，倍數 1.20 → 36 天

## 7. 執行模擬

### 模擬控制參數

在左側邊欄的「Simulation Controls」區段：

#### 迭代次數（N）

| N 值 | 適用場景 | 執行時間（50 活動）|
|------|----------|-------------------|
| 1,000 | 快速探索，結果粗略 | < 0.5 秒 |
| 5,000 | 日常分析 | < 1 秒 |
| **10,000** | **建議值**，結果穩定 | 1-2 秒 |
| 50,000 | 高精度報告 | 5-10 秒 |
| 100,000 | 基準測試驗證 | 10-20 秒 |

> 更多迭代 = 更低的蒙地卡羅標準誤，但收益遞減（誤差 ∝ 1/√N）。P50 通常在 N=5,000 就很穩定；尾部百分位（P90/P95）在 N=10,000 以上才穩定。

#### 取樣方法

| 方法 | 說明 | 建議使用場景 |
|------|------|------------|
| **Pure Random（純隨機）** | 標準蒙地卡羅，每次迭代獨立均勻取樣 | 一般分析 |
| **Latin Hypercube (LHS)** | 分層取樣，確保覆蓋整個機率空間 | 迭代預算有限時；高維問題 |

LHS 優點：在相同迭代數下，通常比純隨機提供更低的估計誤差（約等效於多 30-50% 的迭代數）。

#### 隨機種子（Seed）

固定種子數（例如 `42`）可確保每次執行產生完全相同的結果，適用於：
- 報告存檔
- 對比不同情境（確保差異來自情境本身，而非取樣隨機性）
- 除錯

修改種子數會產生不同但等效有效的結果。

#### 百分位選擇

預設顯示 P50、P80、P90。可以新增任意百分位（P10、P25、P75、P95 等）。

## 8. 結果解讀：S 型曲線與直方圖

### S 型曲線（CDF）

S 型曲線（累積分布函數）顯示「在指定工作天數內完工的機率」。

**讀圖方式**：
- **橫軸**：工作天數（從項目開始）
- **縱軸**：完工機率（0-100%）
- **虛線**：P50/P80/P90 標記
- **灰色點虛線**：CPM 完工日（括號內顯示對應機率）

**典型解讀範例**：

```
CPM 完工：130 天（CPM 線顯示：只有 12% 的機率可以如期完工）
P50：146 天（50% 機率可以在 146 天內完工）
P80：162 天（需要約 32 天緩衝才有 80% 把握）
P90：175 天（合約風險緩衝建議）
```

**向業主溝通**：

> 「CPM 顯示項目在 130 天完工，但根據風險模型，這個日期只有約 12% 的機率可以達成。若要有 80% 的信心，需要至少規劃 162 天。這個差距主要來自 L5 IST 匯合節點的合併偏差——我們稍後在 Merge Bias 分頁有更詳細的分析。」

### 標記目標日期

在「Mark target date」輸入框輸入合約完工日（工作天），S 型曲線會自動標記該日期對應的達成機率。

### 直方圖（PDF）

直方圖顯示所有迭代的完工天數分布形狀。

- **鐘形分布**：路徑活動較多，受中央極限定理影響趨向正態
- **右偏分布**：受少數高風險活動主導，有顯著的悲觀尾部
- **雙峰分布**：可能存在兩條相互競爭的關鍵路徑

## 9. 結果解讀：關鍵性指數與龍捲風圖

### 關鍵性指數（Criticality Index，CI）

**定義**：某活動在所有迭代中，位於最長路徑（最悲觀情境下的要徑）上的比例。

**與 CPM 浮動時間的差異**：

| 衡量指標 | 說明 | 局限性 |
|----------|------|--------|
| CPM 浮動時間 | 靜態計算：晚開始幾天才會影響完工 | 假設所有活動使用確定性（眾數）時程，忽略不確定性 |
| **關鍵性指數（CI）** | **動態計算：實際上有幾趴的機率是要徑** | 需要蒙地卡羅模擬 |

**Hulett (1996) Case 3 的教學案例**：

```
Path A（CPM 要徑）：CPM = 130 天，零浮動
Path B（近要徑）：CPM = 125 天，5 天浮動

模擬結果：
  Path A 關鍵性指數 = 37%
  Path B 關鍵性指數 = 63%  ← 浮動較多的反而更危險！

原因：Path B 活動的分布比 Path A 更為偏態，
      5 天的 CPM 浮動被大幅超越的機率超過 60%。
```

這證明了：**「近要徑」活動（有 CPM 浮動）在實際風險中往往比「CPM 要徑」活動更危險。**

### 顏色編碼

- **紅色柱子**：該活動位於確定性 CPM 要徑上
- **藍色柱子**：該活動不在 CPM 要徑上

若藍色柱子的 CI 高於紅色柱子，代表 CPM 低估了近要徑的風險。

### 龍捲風圖（Tornado Chart）

龍捲風圖以 Spearman 等級相關係數衡量每個活動的持續時間對完工總時程的影響程度。

**解讀**：
- 長度越長 = 對完工日期的影響越大
- 上方的活動是風險緩解的優先目標
- 正值 = 活動越長，整體完工越晚（通常所有活動都是正值）

**明細表格**欄位說明：

| 欄位 | 說明 |
|------|------|
| Criticality Index | 出現在要徑上的機率 |
| Sensitivity (Spearman) | 活動持續時間與完工天數的秩相關係數 |
| CPM Duration | CPM 使用的確定性持續時間（眾數） |
| Simulated Mean | 模擬樣本的平均持續時間 |
| On CPM Critical Path | ★ = 在 CPM 要徑上 |

## 10. 結果解讀：合併偏差

### 什麼是合併偏差？

當多條平行路徑在同一節點匯合時，匯合節點的最早開始時間是：

$$T_{\text{merge}} = \max(X_1, X_2, \ldots, X_k)$$

CPM 計算的是：

$$T_{\text{CPM}} = \max(E[X_1], E[X_2], \ldots, E[X_k])$$

由 **Jensen 不等式**（max 函數是凸函數），可以嚴格證明：

$$E[\max(X_1, \ldots, X_k)] \geq \max(E[X_1], \ldots, E[X_k])$$

因此：**蒙地卡羅模擬預測的完工日期，一定晚於 CPM 的確定性預測**。差距就是「合併偏差」。

### 合併偏差如何增長？

合併偏差隨以下因素增大：
- **平行路徑越多**：每增加一條路徑，都帶來新的「可能成為最遲那條」的機率
- **每條路徑的變異數越高**：分布越寬，尾部越厚
- **路徑間的相關性越低**：完全獨立的路徑比強相關路徑產生更大的偏差

### 資料中心調試的場景

在資料中心/任務關鍵型設施的調試流程中，L5 IST（Integrated Systems Testing）必須等待所有子系統（電力、冷卻、發電機、BMS）分別完成 L3 PFT 和 L4 FPT 後才能開始。

這就是一個 4 條以上平行路徑匯合的典型場景，也是為什麼「現場調試排程幾乎必然比 CPM 計畫晚」的物理原因。

### 合併偏差卡片

| 指標 | 說明 |
|------|------|

| Simulated Mean | 模擬完工的平均值 E[max(...)] |
| Simulated Median (P50) | 完工的中位數 |
| Merge Bias | 模擬均值 − CPM（工作天） |

## 11. 結果解讀：風險驅動因子與相關矩陣

### 相關矩陣熱力圖

熱力圖顯示各活動持續時間之間的 **Pearson 相關係數**（從模擬樣本計算）。

- **深紅色（接近 +1.0）**：強正相關——這兩個活動往往同時延長
- **深藍色（接近 −1.0）**：強負相關（在純風險驅動模型中罕見）
- **白色（接近 0）**：幾乎不相關

### 相關性的物理意義

在 Converge 的模型中，相關性是結構的輸出，不是輸入：

**如果兩個活動共享一個驅動因子**：
- 當驅動因子發生時，兩者同步延長 → 正相關
- 共享越多驅動因子 → 相關性越高
- 每個活動獨有的驅動因子越多 → 稀釋相關性

**與全局相關係數的對比**：

傳統方法是在工具中手動輸入相關係數矩陣，這有兩個問題：
1. 每對活動都要估一個係數（50 個活動 = 1,225 個係數）
2. 係數是主觀的，難以向業主解釋為什麼兩個活動的係數是 0.65 而不是 0.70

風險驅動因子法：你只需要描述「哪些風險事件」會「同時影響哪些活動」，相關性自動湧現，可以完全解釋給非技術受眾。

## 12. 驗證基準台

驗證台是 Converge 的「信任核心」。它包含三層獨立的數學驗證。

### 第一層：分析解一致性驗證

**測試**：對 Triangular(40, 50, 100) 執行 50,000 次迭代，驗證模擬均值和方差與解析公式的吻合度。

**標準**：
- 均值差異 < 4 個標準誤（中央極限定理的統計容忍度）
- 方差差異 < 5%

點擊「Run Analytic Agreement Test」按鈕，通過顯示 **PASS**，失敗顯示 **FAIL** 及數值。

### 第二層：合併偏差 vs 數值積分

**測試**：對兩條獨立三角分布路徑，用 `scipy.integrate.quad` 計算 E[max(X₁, X₂)] 的精確數值，並與模擬結果比較。

**關鍵公式**：

$$E[\max(X_1, X_2)] = \int_{x_{\min}}^{x_{\max}} \left[1 - F_1(x) \cdot F_2(x)\right] dx + x_{\min}$$

**標準**：模擬結果與數值積分差距 < 0.5 工作天。

### 第三層：收斂率驗證（σ/√N）

**測試**：在 N = 500, 1000, 2000, 5000, 10000 各跑 20 次獨立模擬，測量各 N 值下均值估計的標準差，繪製 log-log 圖。

**理論預測**：log-log 斜率 ≈ −0.5（即 SE ∝ N^{−1/2}）

**標準**：實測斜率在 [−0.65, −0.35] 範圍內。

> 這是最強的正確性主張：如果取樣有系統誤差、隨機數生成器有問題，或向量化計算有 bug，這個斜率就不會是 −0.5。

### Hulett (1996) 基準案例

點擊「Run Hulett (1996) Benchmark Cases」，Converge 自動執行三個 Hulett 1996 論文案例並與發表值對比。

| 案例 | 測試目標 | 發表值 |
|------|----------|--------|
| Case 1 | P(CPM ≤ 130d) | 10–15% |
| Case 2 | P(CPM ≤ 130d) | < 5% |
| Case 3 | Path B 關鍵性指數 | ≈ 69% |

## 13. 分布比較：PERT 與三角分布

「**Distribution Comparison**」分頁讓你親眼看到 PERT 與三角分布在相同三點估計下的差異。

### 操作方式

輸入 a（最小）、m（最可能）、b（最大）三個值，頁面自動顯示兩個分布的機率密度疊加圖。

### 關鍵認識：PERT 低估尾部風險

對稱（或微偏態）的三點估計，PERT 和三角分布差異不大。但對高度偏態的活動（例如 a=10, m=12, b=60），差異顯著：

```
Triangular(10, 12, 60): std ≈ 10.9 天
PERT(10, 12, 60):       std ≈  8.3 天  ← 低估了 24%！
```

PERT 有效地強制 std ≈ (b−a)/6，無論偏態程度，因此在極度偏態的活動上系統性低估長尾出現的機率。

**實務建議**：
- **PERT**：適合向業主報告（與業界慣例一致；圖形更平滑）
- **Triangular**：適合保守估計或當歷史資料顯示長尾確實存在時

## 17. 進階：取樣方法（純隨機 vs LHS）

### 為什麼 LHS 更有效？

純隨機蒙地卡羅的問題：雖然在理論上均勻分布，但有限的 N 次迭代下，某些機率區間可能過度取樣，某些可能取樣不足（聚集效應）。

拉丁超立方體取樣（LHS）的解法：
1. 將 [0,1] 分成 N 等分的層（stratum）
2. 每一層內隨機取一個樣本
3. 對每個維度獨立隨機排列

效果：保證每個機率區間各取樣一次，消除聚集效應。

### 在 Converge 中的對比

在「Validation Bench」分頁的收斂率測試中，可以對比 LHS 和純隨機的標準誤：

```
N = 2000 時典型結果：
  Pure Random SE ≈ 0.180
  LHS SE          ≈ 0.145
  → LHS 等效約 2,200 次純隨機迭代
```

### 何時優先使用 LHS？

- 迭代預算有限（N < 5000）
- 活動數量較少（分布維度低時 LHS 效益最大）
- 製作學術報告或需要低估計誤差的場合

## 18. 數學背景：核心概念速查

### 三角分布的解析矩

$$\text{Mean} = \frac{a + m + b}{3}$$

$$\text{Variance} = \frac{a^2 + m^2 + b^2 - am - ab - mb}{18}$$

### PERT 分布的均值

$$\text{Mean}_{PERT} = \frac{a + \lambda m + b}{\lambda + 2}, \quad \lambda = 4 \text{（預設）}$$

$$\text{Variance}_{PERT} = \frac{(\mu - a)(b - \mu)}{\lambda + 3}$$

### Jensen 不等式（合併偏差的根源）

對凸函數 $f$ 和隨機變數 $X$：

$$f(E[X]) \leq E[f(X)]$$

$\max$ 是凸函數，因此：

$$\max(E[X_1], \ldots, E[X_k]) \leq E[\max(X_1, \ldots, X_k)]$$

差距 = 合併偏差 = Converge 的 Merge Bias 卡片顯示的數值。

### 蒙地卡羅標準誤

$$SE_N = \frac{\sigma}{\sqrt{N}}$$

這意味著：
- 迭代數增加 4 倍 → 誤差減少一半
- log-log 圖斜率 = −0.5

### Spearman 等級相關係數（敏感性）

活動 $i$ 的敏感性 = $\text{Spearman}(d_i, T_{completion})$

其中 $d_i$ 是第 $i$ 個活動的持續時間樣本序列。

選用 Spearman 而非 Pearson 的原因：對非線性關係（例如，某活動只在特定情境下成為要徑）也有效。

## 19. 常見問題（FAQ）

**Q：為什麼 CPM 完工日期只有很低的達成機率（例如 10%）？**

A：CPM 使用每個活動的眾數（最可能）時程。但實際上，每個活動都可能比眾數更長。當路徑上有多個活動，且每個活動有獨立的隨機性，整條路徑完工比預期晚的機率相當高。再加上合併偏差，CPM 日期只有 10-20% 達成機率是很正常的。

---

**Q：模擬結果每次都有些不同，這正常嗎？**

A：正常。蒙地卡羅是隨機算法，每次使用不同的隨機種子都會有微小變動。若要完全一致的結果，固定種子數（Seed）即可。若結果波動很大，表示迭代數不足，請增加 N。

---

**Q：我可以在 Converge 中設定「最早開始日期」或「強制完工日期」嗎？**

A：不行，這是**刻意的設計**。強制日期限制在模擬中會讓引擎在每次迭代中「強制成功」，導致機率分布失真，完全無法反映真實風險。風險分析的目的是測試計畫日期是否可行，而不是假設它必然可行。

---

**Q：風險驅動因子的衝擊倍數（impact）只能大於 1.0 嗎？**

A：在 Converge v1 中是的。驅動因子模型假設風險只會延長活動（乘數 ≥ 1.0）。若你的 `impact_a` 設為 1.0，代表風險發生時的最好情況是「無影響」，最壞情況是 `impact_b` 倍延長。

---

**Q：我有 50 個以上的活動，Converge 能處理嗎？**

A：可以。引擎使用 NumPy 向量化，50 個活動在 10,000 次迭代下通常在 2-5 秒內完成。100 個活動也在合理時間內。若速度太慢，可以先用 N=2000 快速探索，確認結果形狀後再用 N=10000 出正式報告。

---

**Q：合併偏差的大小受什麼因素影響最大？**

A：主要因素：
1. **平行路徑數量**：4 條平行路徑比 2 條產生更大偏差
2. **每條路徑的變異數**：分布越寬（b-a 越大），偏差越大
3. **路徑間的相關性**：若路徑完全相關（同一個風險因子影響全部），偏差降低；若完全獨立，偏差最大

---

**Q：如何解讀「Sensitivity (Spearman) = 0.85」？**

A：該活動的持續時間與整體完工日期的 Spearman 等級相關係數為 0.85——這是很高的正相關，意味著這個活動越長，完工日期幾乎必然越晚。這通常是最重要的風險緩解目標。

---

**Q：Converge 可以做成本風險分析嗎？**

A：暫不支援。v1 版本專注於排程風險（時程）。成本風險整合（integrated cost-schedule）列在未來版本的工作清單中。
